JSH

10/2020-12/2020

As part of our Sotware Engineering module, I was assigned to a group of 3 and tasked to refactor and extend an existing implementation of a Java shell called JSH. The previous version contained only one class called JSH which did parsing and could execute only a few applications such as ls, echo, head, tail etc.

The first thing our group discussed was about what development style we were going to use. We settled on using Agile development where we would hold meetings every Sunday to discuss what we had implemented in the previous six days and if there were any issues. We then would plan the next week and made sure that by next sunday we had a working prototype. We continued this cycle every week until we had completed it.

The next main thing we focused on was refactoring. The previous JSH had very bad code design since it had only one class that did everything. It also implemented no design patterns either. So first we partitioned the existing code into different classes and came up with a class structure. JSH would be the main class that runs the code. OurParser handles all parsing and command substitution. We decided to make a super class called Visitable and each application (such as ls, echo, head, tail etc) would extend this superclass. The reason we called it Visitable is because we decided to use the Visitor design pattern here. We also created another class called Call which would basically use the Factory design pattern to create the application object then by using dynamic dispatch we can call visit on the application object without knowing its type. VisitorApplication holds the implementations of each application.

After refactoring, we added more functionality to JSH. We added command substitution, redirection, piping and a few more applications such as cut. This was a lot easier than refactoring as it was completely new code so we could write it in our own style while also being able to focus on just implementing the feature rather than worry about how it would impact the design pattern. While doing this I learnt more about functional Java programming. In particular, I gained some experience using streams. Streams are a higher level abstraction of for loops and helped us tidy up messy long loops into single line statements that would call a method on each stream element. This made the code a lot more readable.

Throughout this project, I was introduced to new software engineering practices such as Test Driven Development(TDD) and Agile while also becoming more familiar with technologies that are used in the software engineering industry. Some of these technologies are Docker and Git. I also learnt about various debugging techniques such as delta debugging and using Visual Studio Code's inbuilt debugging tool.

The hardest part about this project was learning how the visitor design pattern would fit in. Previously, I had never used any design patterns in my code so it took me a while to learn about it in depth and apply it to JSH. Our finished version used several design patterns such as Visitor and Factory while also implementing some more functionality (for example command substitution) into JSH.
